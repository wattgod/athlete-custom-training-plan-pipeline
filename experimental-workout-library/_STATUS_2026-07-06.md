## What this is
A programmatic engine that treats a workout as a **spec** (warm-up template + a main
set of named parameters) and builds `.zwo` files by turning **named physiological
axes**. Every target is % of FTP as a Coggan-zone **range** (Z6/Z7 stay single). Ships
with a 39-archetype × 6-level progressive library plus three earlier "packs."

**Totals:** 329 `.zwo`, all validated (TrainingPeaks-safe: single-quote XML decl,
2/4-space indent, self-closing Warmup/SteadyState/Cooldown, textevents only in
IntervalsT, power 0.30–2.50, parses as XML).

## Layout
```
build_zwo.py                 renderer (the TP-safe .zwo writer + Coggan range band())
progression_engine.py        the engine: axes, warm-up templates, dose (TSS/kJ/W'bal/T@VO2), describe()
archetypes.py                the catalog -> library/  (39 archetypes x 6 levels)
flatten.py                   builds the flat TrainingPeaks-labeled copy
generate*.py                 the 3 earlier bespoke packs (trends, torque, torque-progressions)
library/<9 categories>/      234 .zwo — the archetype library (LIBRARY.md = catalog)
progressions/                28 — trend ladders   |  07 trend singles at root
torque-pack/ (+progressions) 40 — torque pack
engine_demo/                 20 — axis-mechanics proof (NOT ridable sessions)
FUNDAMENTALS_interval-manipulation.md   the named lever taxonomy (Buchheit-Laursen etc.)
WARMUPS.md · MASTER_PLAN.md · RATIONALE.md · README.md
```
The flat upload copy lives OUTSIDE the repo at `~/Downloads/Cycling-Workouts-ALL/`
(309 ridable .zwo, regenerate with `python3 flatten.py` — it writes to that path).

## The journey (how we got here, in order)
1. **YouTube/web research → 7 trending workouts** (Rønnestad 30/15, Seiler 4×8, Kolie
   Moore TTE, over-unders, Almquist Z2+sprints, durability, big-gear torque) → `.zwo`
   built to the house ZWO v6.0 format.
2. **4-level progressions** for those 7 → `progressions/`, + an 8-week polarized plan.
3. **"Not old-hat" push → a distinct torque pack** (8 sessions: FasCat MTI, SFR, EVOQ
   TorqueMax, EF/Rüegg, UAE/Pogačar, CTS stomps, Rouleur sit-stand, descending-cadence)
   → `torque-pack/`, then 4-level ladders with proper warm-ups.
4. **"Make it programmatic" → the engine.** Researched the *fundamentals of interval
   manipulation* (Buchheit & Laursen 9 variables, Seiler, Skiba W′bal, Coggan) → named
   axis taxonomy (`FUNDAMENTALS`) → `progression_engine.py` with TSS/kJ/W′bal/T@VO2max
   computed from the same segments it renders.
5. **Warm-up template library** (8 templates, priming science) auto-matched to main set.
6. **The archetype library** — 39 archetypes across 9 categories, each a ladder.
7. **Titles/labels** iterated for TrainingPeaks grouping: `Group - Name - <level> - mins
   - RPE`, leading with the group word (never a number) so a progression's levels sort
   together; level as a plain number.
8. **Descriptions stripped of justification** (no PURPOSE/PROGRESSION/SOURCE — "the lady
   doth protest too much"); instructions only + a `HOW TO RIDE IT` cue + `FUEL`.
9. **Coggan zones + ranges** — every target became a %FTP range band on the classic 7
   levels; Z6/Z7 (sprint/accel) stay single targets. Encoded as `Power`+`PowerLow/High`
   (and `OnPowerLow/High`) so Zwift holds the flat number and TP reads the band.
10. **Warm-up sanity** — aerobic work (endurance/tempo/sweetspot/durability) now gets a
    10-min Z1→Z2 lift only, no primer; only hard work gets a real warm-up.
11. **Dimensions (Rule #15)** — added a DIMENSIONS line (power zone · cadence · position
    · terrain · effort pattern) to every workout; zones/cadence auto-derived.
12. **Interval set structure** — the micro/high-rep & over-under archetypes were split
    from one continuous block into proper **series** with between-set recovery (Billat,
    Rønnestad 40/20, Tired 30/30, criss-cross ×2, float sets, W-prime).
13. Moved here + committed (this file), 2026-07-06.

## Key design decisions (and why)
- **Axes, not one difficulty knob.** Progression turns ONE named lever (intensity,
  work_duration, recovery_duration, recovery_intensity, reps, sets, density, cadence,
  prefatigue_time/kj, over_ceiling, surge_intensity, shape) and holds the rest.
- **FTP-independent by default.** TSS and the zone ranges need no FTP; kJ and W′bal use a
  250 W *reference* only for the engine demo — they are NOT baked into workout files.
- **`.zwo` range trade-off.** ZWO has single targets or ramps; the flat `Power` +
  `PowerLow/High` pair is the documented range idiom (TP/intervals.icu read it as a band;
  pure Zwift honors the flat `Power`). Verified against the h4l ZWO tag reference.

## Open items / NOT done
- **No pipeline integration.** Not called by `intake_to_plan.py`; no per-athlete FTP, no
  W′bal gating, no calendar/date-stamping, no methodology selection. The obvious next step
  is to expose the engine to `build_block` so ladders render per-athlete with a real FTP.
- **Two description styles.** Library descriptions are auto-derived (zones/RPE/dimensions);
  the trend/torque packs keep hand-written prose with dimensions injected at flatten time.
- **L5–L6 are elite.** The top rungs get large by design (e.g. Billat L6 ≈ 3×11). Treat as
  aspirational, or switch those archetypes' progression axis to grow sets not reps.
- **Named primary studies** are attributed via secondary sources in places — verify DOIs
  before any of this becomes published Gravel God / Endure content.
- **Not deduped against the house catalog.** This mirrors it conceptually but was built
  fresh; if it graduates, reconcile with `archetypes/` + `athletes/scripts/*archetype*.py`.
